# 医学统计与数据科学方法字典

这是一个面向医学研究、临床预测、流行病学和生物医学数据分析的 Markdown 知识库。目标不是只罗列“方法名称”，而是把每个方法整理成可直接用于学习、选型、实现和解释结果的方法卡片。

这个字典希望同时回答 4 个问题：

- 这个方法在解决什么问题
- 这个方法背后的数学形式是什么
- 这个方法在医学研究里什么时候该用、什么时候不该用
- 这个方法如何用代码实现、如何解释结果、如何可视化

## 项目定位

建议把这个项目建设成一个“方法词典 + 选型手册 + 实现备忘录”的结合体。

- 对学习来说，它帮助你把概念、公式、直觉和代码放在同一页里
- 对实战来说，它帮助你从研究问题快速映射到方法
- 对写作来说，它帮助你沉淀结果解读、图表表达和常见陷阱

适合覆盖的内容包括：

- 经典医学统计
- 回归建模与广义线性模型
- 生存分析与纵向数据分析
- 因果推断
- 树模型、聚类、降维、时间序列与高维建模
- 分类与判别方法
- 模型评估、校准、解释与可视化

## 方法卡的统一框架

建议每个方法/模型都按同一套更紧凑的结构编写。当前推荐用 12 个主标题：

1. 方法概览
2. 数学形式
3. 数据形式与输入输出
4. 适用场景
5. 实现
6. 结果如何解释
7. 推荐可视化
8. 优势、局限与常见坑
9. 与相近方法的区别
10. 医学研究中的典型应用
11. 相关方法
12. 参考资料

其中前 3 个主标题建议重点包含这些信息：

### 1. 方法概览

- 方法中文名与英文名
- 一句话定义
- 它主要解决的研究问题
- 直觉理解

### 2. 数学形式

- 核心公式
- 参数或统计量的含义
- 关键假设

### 3. 数据形式与输入输出

- 适合的数据形式
- 自变量和因变量的类型
- 示例表格
- 输入数据、关键变量、预处理内容
- 模型产出、参数估计、预测结果、不确定性指标

## 推荐分类框架

当前更推荐按“具体方法家族”来组织，而不是设置一个过于宽泛的 `机器学习` 总目录。这样更适合后续做网站导航，也更方便读者按问题类型直接定位。

1. 研究设计与数据理解
2. 描述统计与统计推断
3. 回归与广义线性模型
4. 生存分析与纵向数据
5. 因果推断
6. 正则化与变量选择
7. 树模型与集成学习
8. 支持向量机与核方法
9. 聚类与无监督学习
10. 降维与表征学习
11. 神经网络与深度学习
12. 时间序列与时序建模
13. 数据预处理与特征工程
14. 高维数据与多组学
15. 模型评估与解释
16. 分类与判别方法

这种划分方式主要借鉴了“按算法家族组织”的思路，但做了适合医学统计与数据科学知识库的调整：

- 保留统计主线：统计推断、回归、生存、因果
- 去掉过于笼统的 `机器学习`
- 把未来最可能扩展的内容拆成更清晰的门类，如树模型、聚类、降维、时间序列、特征工程
- 对于 K 近邻、朴素贝叶斯、LDA/QDA 一类以“判别”为主、又不完全属于回归或树模型家族的方法，单独留出 `分类与判别方法`

## 编写规范

### 命名建议

- 一个方法一篇笔记
- 同一目录内建议按阅读顺序加前缀编号，文件名建议使用 `NN-中文名（英文名）.md`
- 首次出现时写全称，后文再用简称

例如：

- `01-Logistic回归（Logistic Regression）.md`
- `02-Cox比例风险模型（Cox Proportional Hazards Model）.md`
- `03-随机森林（Random Forest）.md`

### 内容规范

- 优先写“这个方法解决什么问题”，再写“它怎么推导”
- 公式要尽量配参数解释
- 代码示例尽量给出 Python 和 R 两个版本，至少保留一个主版本
- 每篇方法卡尽量写出“适用场景”和“不适用场景”
- 每篇都保留“结果解释”和“可视化建议”这两个部分
- 每篇末尾尽量给出简洁的外部参考资料，优先使用教材、方法论文和官方文档，不把本地笔记作为正式参考文献

### 医学场景补充建议

对于医学统计方法，尤其建议额外记录：

- 结局类型：连续型、二分类、计数型、时间到事件、多分类、重复测量
- 缺失机制：MCAR、MAR、MNAR
- 是否涉及删失
- 是否需要考虑混杂、选择偏倚、类别不平衡
- 是否强调可解释性、可校准性、可外部验证

## 案例数据资产

`N个数据分析案例` 的主要价值不在于直接复用整套分析代码，而在于把其中的数据集沉淀成这个知识库的“可复用演示数据资产”。

后续在新增方法卡时，优先考虑复用这些数据集来做：

- 数据表结构示例
- 变量类型示例
- 清洗前后对比示例
- 可视化图像示例
- Python / R 最小可运行脚本示例

### 复用原则

- `N个数据分析案例/code` 中的脚本只作为理解数据集与分析思路的参考，不作为本知识库的方法实现规范。
- 本知识库新增方法卡时，应优先重写“最小、清晰、可解释”的演示代码，而不是直接复制案例脚本。
- 复用重点放在“数据形态演示”和“图表演示”，而不是完整复刻原项目。
- 若案例数据不适合某方法的最小示例，可以补充使用 Python 和 R 的内置数据集。
- 如果案例标题与真实数据结构不完全一致，应以数据文件本身的结构为准。
- 如果案例文档中的相对路径、清洗步骤或建模流程与这里的整理方式不完全一致，应以当前知识库中的数据文件和方法卡写作目标为准。

### 重点数据特点

#### 1. 时间序列与面板型数据

- `GlobalTemperatures.csv` 适合演示单变量时间趋势、多变量同步趋势、缺失值插补、滚动窗口平滑。
- `all_stocks_5yr.csv` 适合演示“`date + group + value`”这种长表结构，尤其适合按股票代码分组的折线图、收益率计算、波动分析。
- 这两类数据非常适合补充到 `描述统计`、`可视化`、`时间序列`、`非参数平滑` 等条目。

#### 2. 宽表监督学习数据

- 房价、客户流失、乳腺癌、信用卡欺诈都属于“每行一个样本、每列一个变量”的经典宽表。
- 其中房价数据是**连续结局回归**的好例子，流失/乳腺癌/欺诈属于**二分类**例子。
- 这类数据非常适合演示：
  - 描述性统计表
  - 缺失值统计表
  - 数值/分类变量混合清洗
  - 训练集/测试集拆分
  - 回归、分类、正则化、模型评估

#### 3. 关系型与半结构化多表数据

- 电影数据集不是单表，而是多个 CSV 组合而成，且 `genres`、`keywords`、`cast`、`crew` 等字段包含类 JSON 字符串。
- 它非常适合演示：
  - 多表合并
  - 一对多关系
  - 嵌套字符串解析
  - 高基数分类变量汇总
  - 大文件抽样分析
- 但它不适合作为“最小方法示例”的默认首选，因为清洗成本明显更高。

#### 4. 类别不平衡与异常检测数据

- `creditcard.csv` 是非常典型的极端不平衡数据：`Class=1` 只有 492 条。
- 这类数据很适合未来在以下方法卡中做演示：
  - 逻辑回归在不平衡数据下的局限
  - Precision / Recall / F1 / PR 曲线
  - 重采样、阈值调整、异常检测
- 但不适合直接作为“箱线图演示”的首选，因为全量作图容易被多数类淹没，通常需要抽样。

#### 5. 文本型数据

- `spam.csv` 清洗前有 5 列，但真正核心列只有标签列和短信文本列，其余 3 列本质上是空白占位列。
- 它非常适合演示：
  - 文本清洗
  - 字符串长度分布
  - 词频统计
  - TF-IDF
  - 朴素贝叶斯 / 逻辑回归文本分类
- 与医学统计主线不同，但对“数据类型扩展”很有价值。

#### 6. 纵向与生存数据

- `Framingham_data.csv` 是这批数据资产中最适合医学统计主线演示的资源之一。
- 它同时支持：
  - 基线横断面分析
  - 重复测量纵向分析
  - 生存结局分析
  - 风险因素与效应修饰演示
- 未来如果要写混合效应模型、GEE、Kaplan-Meier、Cox 比例风险模型，这个数据集应该作为优先演示数据。

#### 7. 标题与真实数据结构不完全一致的案例

- `007-基础的癌症生存率数据探索` 按当前数据文件内容看，实际更接近**乳腺癌良恶性分类数据**，并不包含真正的生存时间和删失结局。
- 因此后续若写“生存分析”方法条目，不建议用这个数据集做主示例，应优先用 `Framingham_data.csv` 或其他真正含 `time + event` 的数据。
